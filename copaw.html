CoPaw 技术架构介绍（2026）

这是一份基于 CoPaw 最新源码的架构说明，聚焦当前真实实现： FastAPI 控制面、Runner 会话执行、AgentScope ReAct 内核扩展、Channel 插件化接入、MCP 热更新、Cron/Heartbeat 调度、Skill 与本地模型生态。

代码基线：CoPaw@cd47a43 更新时间：2026-03-04 范围：CLI + App + Agent + Channels + MCP + Cron
一句话概括： CoPaw 是“多渠道接入控制面 + AgentScope ReAct 执行内核 + 可热更新外设（MCP/Channel）+ 调度化运行”的个人 AI 助手平台。

多 Agent 支持现状

结论（截至 2026-03-04）：CoPaw 当前开箱主链路以单 Agent 会话执行为主，产品级多 Agent 仍在 roadmap（Long-term Planning）。

  • 当前主执行模型是 Runner + CoPawAgent(ReAct) 的单会话路径。
  • 官方 roadmap 中 Multi-agent 明确标注为 Long-term Planning。
  • 因其底座来自 AgentScope，后续具备承接多 Agent 能力的架构基础，但当前产品面未作为已完成能力交付。

架构总览与分层

Layer A

入口与交互层

CLI 入口、FastAPI 服务、Console SPA、管理 API 与多聊天渠道接入。

Layer B

控制面与编排层

Runner、会话状态、配置热更新、MCP 客户端生命周期、定时任务编排。

Layer C

Agent 执行层

基于 AgentScope ReAct 的 CoPawAgent，负责推理循环、工具调用、命令处理与消息流式输出。

Layer D

能力与策略层

内置工具、Memory 管理、Bootstrap/Compact Hook、Skills 生命周期与系统提示装配。

Layer E

模型与扩展生态层

Provider 注册、本地模型下载管理、Ollama/MLX/llama.cpp 兼容与外部 MCP 生态接入。

与 OpenClaw 子系统对照关系

CoPaw 在产品形态上与 OpenClaw 同属“可接入多渠道的 Agent 控制面”，但协议与实现路径不同。 下列映射用于横向对比职责，而非一一同构。

OpenClaw 01/02/03/04

Gateway 控制面主链路

对应 CoPaw： 02 FastAPI 应用内核 03 管理 API 04 Runner 10 Channel Manager

OpenClaw 05/07/08/09

Agent 能力执行链

对应 CoPaw： 05 CoPawAgent 06 工具执行层 07 Memory 09 Skills

OpenClaw 06

外部 Agent 协议控制面

OpenClaw 是 ACP 控制平面；CoPaw 的近似角色是 11 MCP 控制平面。 协议不同，但都承担“外部能力/Agent 接入编排”职责。

OpenClaw 10/14

多端接入与调度

对应 CoPaw： 10 Channel 插件接入层 13 Cron/Heartbeat 调度 03 Console/API 管理面

OpenClaw 11/12/13

安全/沙箱/浏览器差异

CoPaw 内置 06 browser_use 工具， 但不像 OpenClaw 一样在核心层内建完整 Sandbox 控制面； 隔离策略通常由部署方式或外部运行时承担。

结论：CoPaw 与 OpenClaw 在“控制面 + Agent 执行 + 扩展接入”三段式上同构，但外部协议和隔离深度不同。

MCP 控制平面深度解析

CoPaw 的 MCP 不是简单“读取一次配置并连接”，而是一个运行时可持续治理的控制面： Manager 负责生命周期，Watcher 负责差异感知与重载策略，Runner 每轮请求动态取最新客户端集合。

MCP-01

职责分离：管理器与监控器

  • MCPClientManager：负责 connect/replace/remove/close_all。
  • MCPConfigWatcher：负责监听配置变更、触发增量更新。

src/copaw/app/mcp/manager.pysrc/copaw/app/mcp/watcher.py

MCP-02

低锁竞争替换策略

  • 先在锁外连接新 client（可设置超时）。
  • 连接成功后再在锁内原子替换并关闭旧连接。
  • 避免慢连接阻塞整个管理器。

src/copaw/app/mcp/manager.py

MCP-03

失败治理与重试熔断

  • Watcher 记录每个 client 的失败次数与配置哈希。
  • 同一配置连续失败达到上限后跳过重试，等待配置变更再恢复。

src/copaw/app/mcp/watcher.py

MCP-04

Runner 动态取客户端

  • 每次 query 时从 manager 拉取当前 client 列表。
  • 新配置无需重启服务即可在后续对话生效。

src/copaw/app/runner/runner.py

MCP-05

Agent 侧注册与断连恢复

  • CoPawAgent 注册 MCP 工具时处理连接中断与取消异常。
  • 通过 _copaw_rebuild_info 重建 client 并尝试重新连接。

src/copaw/agents/react_agent.pysrc/copaw/app/mcp/manager.py

MCP-06

API 管理面与敏感信息保护

  • 通过 /api/mcp 提供客户端增删改查。
  • env/header 在返回时做掩码处理，降低凭据泄漏风险。

src/copaw/app/routers/mcp.py

结论：CoPaw 的 MCP 控制平面本质上是“配置驱动的在线连接编排器”，它保证 MCP 生态可以在服务运行中持续演化，而不牺牲主链路稳定性。

关键运行链路

Flow A

消息处理主链路

  • Channel/Console/HTTP 输入进入 Runner。
  • Runner 构建 env_context + 注入 MCP clients + 创建 CoPawAgent。
  • Agent 走 ReAct 循环，流式输出结果并持久化 session/chat。
Flow B

渠道并发与防抖链路

  • ChannelManager 按 channel 建队列，按 debounce_key 聚合同 session 消息。
  • 同 session 串行，跨 session 并行，处理中新增消息进入 pending 合并。
Flow C

Cron/Heartbeat 链路

  • CronManager 加载任务并注册调度器。
  • 触发时经 Executor 调用 Runner 或 Channel 发送任务结果。
  • ConfigWatcher 检测 heartbeat 变更后执行重调度。

工程特征

  • 控制面在线化：MCP、Channel、Heartbeat 支持运行中动态调整。
  • 执行链收敛：统一 Runner + Agent 抽象，减少多入口行为分叉。
  • 能力可组合：Toolkit + Skills + MCP 形成统一工具扩展面。
  • 会话可恢复：JSON Session + Chat 元数据可追踪、可重放。
  • 本地优先可控：本地模型、工作目录、技能目录全部在用户侧自治。

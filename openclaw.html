OpenClaw 技术架构介绍（2026）

这是一份基于 OpenClaw 最新源码的架构说明，聚焦系统当前的真实实现： 网关控制面、Agent 双运行时、工具策略管线、记忆系统、通道插件化、Sandbox 隔离、Browser 自动化、Node/Device 能力与多端体验。

代码基线：openclaw@ad9ceafec 更新时间：2026-03-04 范围：Gateway + Runtime + Extensions + Apps
一句话概括： OpenClaw 已从“单一聊天网关”演进为“多入口控制平面 + 双 Agent 运行时 + 插件生态 + 多端节点能力网络”的完整平台架构。

多 Agent 支持现状

结论（截至 2026-03-04）：已支持且为当前能力，不是 roadmap。

  • 支持按渠道/账号/会话进行多 Agent 路由与隔离（workspace + session 维度）。
  • 主链路同时包含 Embedded Pi Runtime 与 ACP Runtime，支持跨会话协同与子 Agent 编排。
  • 具备 Agent-to-Agent 会话工具能力（如 sessions 系列工具），可在不同 Agent 会话间传递任务与上下文。

架构总览与分层

Layer A

入口与交互层

WebSocket RPC、HTTP API（OpenAI/OpenResponses）、Control UI、WebChat、Canvas/A2UI、CLI、多端 App。

Layer B

控制面与路由层

协议校验、方法权限、会话路由、并发 lane 调度、事件分发、健康与状态管理。

Layer C

Agent 执行层

Embedded Pi Runtime（主路径）+ ACP Runtime（控制面会话），共同驱动自动回复与编排执行。

Layer D

能力与安全层

Tool Pipeline、Memory、Skills、Sandbox、Browser、DM/Group Policy、Secrets、安全审计与修复。

Layer E

扩展与节点层

插件运行时（channel/provider/tool/hook/http/service）、Node Host、设备配对与远程能力调用。

ACP 控制平面深度解析

在 OpenClaw 中，ACP 不是单一模块，而是两条协同路径： 协议桥接路径（IDE/客户端 ↔ Gateway）和 运行时控制路径（Gateway ↔ 外部 ACP Runtime Backend）。

ACP-01

协议定位：Bridge 与 Runtime 双路径

  • Bridge 路径：openclaw acp 通过 ACP stdio 协议接 IDE，再用 WebSocket 转发到 Gateway。
  • Runtime 路径：/acp spawn 或 sessions_spawn(runtime="acp") 由 Gateway 调度外部 ACP 兼容 Agent。

src/acp/server.tssrc/acp/translator.tssrc/agents/acp-spawn.ts

ACP-02

控制面与后端分离

  • 控制面（Gateway 内）：会话元数据、状态机（idle/running/error）、生命周期（init/run/cancel/close）、策略与路由。
  • 后端执行层（插件）：运行时后端通过 registry 注册（默认 acpx），实际执行由后端实现。
  • acpx 实现：插件服务注册 backend，运行时通过宿主机命令调用外部 CLI。

src/acp/control-plane/manager.core.tssrc/acp/runtime/registry.tsextensions/acpx/src/service.tsextensions/acpx/src/runtime.ts

ACP-03

串行化编排：Session Actor Queue

  • 按 sessionKey 归一化为 actor key，所有 turn/cancel/close 操作进入同一串行队列。
  • 并发治理包含：活跃 turn 注册、AbortController 取消、队列深度统计。
  • 配合 runtime cache + TTL 驱逐 + maxConcurrentSessions，避免会话无限堆积。

src/acp/control-plane/session-actor-queue.tssrc/acp/control-plane/runtime-cache.tssrc/acp/control-plane/manager.core.ts

ACP-04

线程绑定与消息路由

  • spawn 请求可触发 thread binding：先校验 channel/account 策略，再绑定 conversation 到 ACP session。
  • 绑定成功后，后续线程消息进入同一 ACP 会话，输出也回投该线程。
  • spawn 失败、手动 close 时会执行 unbind 清理，避免悬挂绑定。

src/agents/acp-spawn.tssrc/auto-reply/reply/dispatch-acp.tssrc/acp/control-plane/spawn.tssrc/channels/thread-bindings-policy.ts

ACP-05

能力协商与运行时选项下发

  • 控制面先读取 backend capabilities，再决定是否允许 session/set_mode 与 session/set_config_option。
  • 会话 runtimeOptions 持久化后，在 turn 前下发到后端并做签名去重，避免重复配置调用。
  • 标准映射：model、approval_policy、timeout、cwd 与 backend extras。

src/acp/control-plane/manager.runtime-controls.tssrc/acp/control-plane/runtime-options.ts

ACP-06

身份对齐与可观测性

  • 会话 identity 采用 pending/resolved 状态，从 ensure/status 合并外部 session identifiers。
  • Gateway 启动时会对 pending 身份执行 reconcile，降低重启后的会话漂移。
  • 内置观测快照：active sessions、queue depth、latency、error code 分布。

src/acp/runtime/session-identity.tssrc/acp/control-plane/manager.identity-reconcile.tssrc/gateway/server-startup.ts

ACP-07

安全边界：Host 与 Sandbox 的明确分界

  • ACP runtime 默认在宿主机侧执行（后端进程/CLI 路径），不在 OpenClaw Docker sandbox 内部运行。
  • 沙箱会话禁止发起 runtime=acp 的 spawn；sandbox="require" 对 ACP 也会被拒绝。
  • 策略闸门包括 acp.enabled、acp.dispatch.enabled、allowedAgents，以及高危 ACP tool 显式审批。

src/agents/acp-spawn.tssrc/acp/policy.tssrc/security/dangerous-tools.tssrc/acp/client.ts

结论：OpenClaw 的 ACP 控制平面本质上是“Gateway 主导的会话控制面 + 可插拔后端执行面”。 它并不与 Embedded Pi 竞争，而是为外部 ACP Agent 提供统一的会话编排、线程路由与安全治理能力。

关键运行链路

Flow A

消息处理主链路

  • Channel/Webhook/WS 输入进入 Gateway。
  • 会话键解析 + policy 检查 + queue 模式决策。
  • 进入 Embedded Pi 或 ACP runtime 执行。
  • tool/memory/skills 协同后输出，回写会话与下行通道。
Flow B

定时与心跳链路

  • CronService 根据调度触发 isolated/main 会话执行。
  • Heartbeat wake 采用合并与重试，避免高频抖动。
  • 结果可走 announce、channel 或 webhook 递送。
Flow C

节点能力调用链路

  • Gateway 维护 Node Registry 与订阅关系。
  • node.invoke.request 下发到 Node Host。
  • Node Host 按执行策略返回 invoke.result 并回灌会话事件。

工程特征

  • 协议先行：schema 驱动 + 统一错误模型，降低跨端实现偏差。
  • 策略内建：权限、沙箱、队列、审计并非外挂，而是主路径组成部分。
  • 运行时双轨：Embedded Pi 与 ACP 并存，兼顾稳定默认路径与可编排扩展能力。
  • 插件优先扩展：渠道、工具、provider、服务都可插件化接入。
  • 端到端闭环：从 Gateway 到 Node/App/UI 的能力是连续系统，不是孤立模块。

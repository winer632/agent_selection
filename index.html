Agent 架构选型分析报告（知识库问答 + 工作流自动化）

版本：v1.0

日期：2026-03-02

面向：组内技术选型评审

一、执行摘要

结论如下：

  1. OpenClaw/Pi 在“开放式自主执行”上智能上限高，但默认范式是 代码生成 + 执行（ACE），不适合作为企业无人值守生产底座。
  2. 对于我们当前主场景（知识库问答 + 工作流自动化），应优先选择 Tool/Function Calling 范式，而非 Bash/脚本直跑范式。
  3. 推荐主路径：
    • 交付效率优先：Dify + 受限工具网关（API/MCP）
    • 复杂流程可控优先：LangGraph + PydanticAI（结构化输出）
    • 多智能体与 Skills 生态优先：AgentScope（或 CoPaw 仅作参考实现）
  4. OpenClaw / Nanoclaw 仅建议用于 PoC/创新实验、个人助手类场景，不建议直接用于企业生产自动化主链路。

二、项目背景与约束

  • 业务目标：客户定制化 Agentic 项目，重点是知识库问答（RAG）与工作流自动化。
  • 非功能目标：安全合规、可审计、可持续迭代、可运维交付。
  • 当前已讨论安全措施：Docker 隔离、固定接口入参、输入过滤、内网数据源（向量库/MariaDB/MinIO）、禁外网搜索、Skills 调用。

三、OpenClaw分析

3.1 OpenClaw + Pi 的价值与边界

  1. OpenClaw 选择 Pi 的核心原因是“极简且稳定的执行内核”与“外围网关能力”互补。
  2. Pi 的 Read/Write/Edit/Bash 极简工具集降低了内核复杂度，适合做长期运行底座。
  3. OpenClaw 通过 SDK 嵌入 Pi，而不是外部进程调用，因此能够接管会话生命周期、权限边界和工具注入。
  4. 在能力定位上，Pi 更偏“自主写代码并执行”，OpenClaw 更偏“渠道接入、记忆、调度与生态封装”。

3.2 OpenClaw既有架构能力拆解

  1. OpenClaw 在 Pi 之上补齐了企业常见“入口与运营层”能力，包括多渠道接入、持久记忆、定时/主动触发、技能生态和网关面板。
  2. Pi 可直接通过 SDK 注入自定义工具，但 SKILL.md 这种可分发技能形态主要是 OpenClaw 上层生态能力。
  3. 从工程视角看，OpenClaw/Pi 的优势是开放式问题求解能力强，弱点是执行路径不可预测。

3.3 OpenClaw安全风险的根因判断

  1. 主要风险是范式风险，即过度授权 + 任意代码执行 + 提示词注入攻击面。
  2. 仅做 Docker 挂载目录、固定入参接口、输入过滤，能降低风险但不足以形成企业级闭环。
  3. 关键缺口包括数据外发、内网横向访问、密钥滥用、资源耗尽以及“黑盒脚本执行难审计”。
  4. 即便只使用 Pi 而不用 OpenClaw，上述高自由执行风险也不会消失，本质是同一执行哲学导致。

3.4 OpenClaw HITL 与自动化体验的平衡

  1. 人类在环（HITL）确认机制对高危动作有效，但不能把“每一步都确认”当作默认方案。
  2. 在 RAG 与流程自动化场景中，逐步弹窗会显著破坏体验，并导致业务用户无法有效审阅代码级确认内容。
  3. 可落地的治理方式是“按工具分级”而非“按步骤一刀切”：低风险读操作自动放行，高风险写操作强制审批。
  4. Claude Cowork / Claude Code 能进入企业场景，核心也在于隔离、结构化工具协议、策略治理和审计链路，而不是“模型天然更安全”。

3.5 代码执行型范式分析

  1. 定义：Agent 在运行时自主生成并执行代码/命令（如 Bash、Python），通过“现场造工具”完成任务。
  2. 优点（智能能力）：
    • 泛化能力高，遇到未预置工具的任务也能尝试完成。
    • 对开放问题有较强探索能力，适合快速 PoC 和个人效率场景。
    • 在长尾任务上常表现出较强“即兴解决问题”能力。
  3. 缺点（安全合规）：
    • 与提示词注入、过度权限叠加后，风险放大明显。
    • 执行路径不可预测，事前审核和事后审计难度高。
    • 无人值守场景下容易出现误操作、越权访问、资源耗尽和内部破坏风险。
  4. 工程代价：
    • 需额外建设强隔离沙盒、最小权限、网络白名单、审批网关和审计回放。
    • 即便防护完备，系统复杂度与运维成本通常仍高于工具调用型方案。
  5. 本项目结论：
    • 代码执行型范式可作为“受控特例能力”，不应作为企业 RAG + 工作流主干。
    • 主路径仍应以可控工具调用为核心，把高风险代码执行限制在强隔离、可审批的边界内。

4.1.1 Agent Teams 选型补充建议

  • 如果核心目标是“多角色协作产出”（如研究员/审核员/执行员分工），优先看 AgentScope、CrewAI、LangGraph。
  • 如果核心目标是“RAG + 业务流程”而非复杂多 Agent 协商，优先 Dify 或 LangGraph + PydanticAI，把 Agent Teams 作为二期能力。
  • 多 Agent 数量建议从 2-3 个角色起步，先做职责边界与失败回退，再扩展团队规模。

4.1.2 代码执行工具清单（重点框架）

说明：以下用于解释“支持代码执行型范式”的具体技术落点。支持不代表必须启用，生产环境建议默认禁用或强隔离。
打开代码执行后，风险会显著上升；是否达到与 OpenClaw 相同风险水平，取决于权限边界、隔离强度与审批审计策略。

框架 代码执行工具/组件（常见） 说明
LangGraph Python REPL、Shell (Bash)、ShellToolMiddleware（HostExecutionPolicy / DockerExecutionPolicy / CodexSandboxExecutionPolicy） LangGraph 是编排层，代码执行能力通常来自 LangChain 工具与 middleware 配置。
AgentScope execute_python_code、execute_shell_command 来自 agentscope.tool API 的内置能力，也可通过自定义工具或 MCP 扩展。
CoPaw （继承 AgentScope）execute_python_code、execute_shell_command CoPaw 底层为 AgentScope，代码执行能力与禁用策略基本一致。
Claude Agent SDK Bash / 终端执行能力、文件编辑/写入工具链（由 SDK 运行时提供） 官方代码代理 SDK，默认就面向“可执行任务”场景，需强治理。
Nanoclaw （继承 Claude Code/Agent SDK）Bash、文件读写/编辑工具链（容器内执行） 把执行环境收敛到容器工作区，但默认范式仍为“代码生成 + 执行”，需审批与审计策略。
LlamaIndex CodeInterpreterToolSpec、PandasQueryEngine（NL 转 Python） 文档明确这类能力涉及执行风险，需沙盒化与权限收敛。
Haystack Tool(function=Callable) + ToolInvoker、ComponentTool Haystack 偏通用工具执行器，是否代码执行取决于你绑定的函数/组件。
CrewAI CodeInterpreterTool、allow_code_execution=True 可用 Docker 沙盒，配置不当时也可落到宿主执行，需强约束。
AutoGen PythonCodeExecutionTool、DockerCommandLineCodeExecutor、LocalCommandLineCodeExecutor、CodeExecutorAgent 官方长期提供代码执行 Agent/执行器体系，默认应优先 Docker/隔离执行。
OpenClaw 底层 Pi 工具原语：Read、Write、Edit、Bash（其中 Bash 为核心执行能力） 默认范式即“代码生成 + 执行”，企业无人值守场景治理成本最高。

五、针对业务场景的推荐架构

5.1 推荐架构

API Gateway / BFF
→ Agent Orchestrator（Dify 或 LangGraph）
→ Tool Layer（只暴露白名单函数，不给 Bash）
→ Data Layer（VectorDB / MariaDB / MinIO）
→ Audit & Policy（RBAC、审批、审计日志）

5.2 关键原则

  1. 禁止 LLM 直接命令执行权限：不把 Bash/代码执行暴露为默认能力。
  2. 读写分级：查询类工具自动放行；外部副作用类操作（写库、发邮件、改工单）必须策略审批。
  3. 最小权限凭证：数据库只读账户、按表按动作授权、短期密钥。
  4. 知识库防注入：文档入库前清洗 + 检索后指令隔离（将文档内容视为不可信数据）。
  5. 网络出口白名单：禁公网泛访问，按域名/IP 出站控制。
  6. 全链路可审计：记录“模型意图 -> 工具调用 -> 参数 -> 结果”。

六、最终选型建议

方案 A（推荐）

  • Dify（前台编排交付） + 受限 Tool Gateway（后端策略）
  • 适合：交付周期短、客户需要可视化运营后台、团队希望降低研发门槛。

方案 B（推荐）

  • LangGraph + PydanticAI（核心编排 + 严格结构化输出）
  • 适合：复杂流程、强审计、需要精细化工程控制的项目。

方案 C（特定场景推荐）

  • AgentScope（必要时结合 CoPaw 思路，不直接搬 CoPaw 全栈）
  • 适合：多 Agent 协同、Skill 体系深度定制、偏 Python 工程团队。

不建议作为企业生产主底座

  • OpenClaw / Pi / Nanoclaw（除非仅在强隔离沙盒内做实验或内部 PoC）
  • Claude Agent SDK / OpenCode（同类代码执行型范式，需极高治理成本）

七、会议讨论清单

  1. 本期项目的主优化目标是什么：优先“更快上线与交付效率”（开发/运维成本更低），还是优先“治理可控与审计完备”（安全与合规约束更强）？
  2. 是否接受“读操作默认免审批、写操作按风险分级审批”的策略（即使不开启代码执行型范式，该策略仍适用于发邮件、写库、改工单等业务写操作）？
  3. 是否先定义“高风险写操作清单”（如外发、删除、资金相关、批量修改），并统一要求审批？
  4. 客户是否要求可视化后台（偏 Dify）还是纯代码可控（偏 LangGraph）？
  5. 是否需要在一期就引入多 Agent 协作（偏 AgentScope）？
  6. POC 阶段是否直接采用 OpenClaw：以“演示智能度和惊艳效果”为第一目标快速验证，但明确与生产交付底座分离（POC 演示路线 vs 生产治理路线）？
